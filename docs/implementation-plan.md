## 管理端隔离方案

当前代码已经先做了一步隔离：

1. 前台导航不再展示管理端入口。
2. `admin` 路径使用独立后台壳层，不复用前台导航表现。
3. 前台页面只保留用户可见的数据展示能力。

后续推荐继续按下面顺序拆分：

1. 路由隔离
   前台保持主站域名，后台改为独立子域，例如 `admin.xxx.com`。
2. 入口隔离
   后台不在前台页面放显式按钮，只通过受控地址和登录后跳转进入。
3. 权限隔离
   后台增加登录态、中间件鉴权、角色权限控制。
4. 部署隔离
   前台与后台可继续同仓，也可以拆成两个 Vercel Project；采集后端仍保持独立服务。

## 管理端详细设计

### 1. 目标定位

管理端不是展示站点，而是内部运营控制台。它只服务管理员和审核人员，核心目标有四个：

1. 管理数据源和采集策略。
2. 管理抓取任务与异常处理。
3. 审核 AI 结构化结果并控制发布。
4. 保证所有发布动作可追踪、可回滚、可审计。

### 2. 访问与隔离策略

推荐按三层隔离设计：

1. 入口隔离
   默认不在前台暴露任何后台入口。
2. 路由隔离
   正式环境使用独立子域，例如 `admin.domain.com`。
3. 运行隔离
   通过环境变量控制后台是否启用；未启用时直接返回 404。

当前代码已按第 3 点做了第一步处理：`ENABLE_ADMIN_UI !== true` 时，`/admin` 直接不可访问。

### 3. 角色设计

建议至少拆成 3 个角色：

1. `viewer`
   只看任务状态、发布记录和日志摘要。
2. `reviewer`
   可处理人工验证、修正 AI 结果、提交审核。
3. `admin`
   可管理数据源、触发抓取、发布数据、回滚版本、修改系统配置。

### 4. 信息架构

后台建议只保留 6 个主模块：

1. 仪表盘
   今日任务数、成功率、待审核数、待人工验证数、最近发布记录。
2. 数据源
   新增站点、编辑站点、启停采集、设置抓取模式、维护备注和标签。
3. 任务中心
   手动抓取、定时任务列表、执行状态、错误摘要、失败重试。
4. 人工验证
   查看需要登录或验证码的任务，处理后继续执行。
5. 审核发布
   左侧原始内容，中间结构化结果，右侧历史对比与差异结论，底部发布动作。
6. 发布记录
   查看版本、差异摘要、发布时间、操作者、回滚入口。

### 5. 页面布局设计

后台布局建议固定为三段式：

1. 左侧导航
   固定宽度，展示模块分组与当前状态徽标。
2. 顶部操作栏
   搜索、筛选、环境标识、账号菜单。
3. 主工作区
   使用表格、任务列、表单和审核分栏，不使用营销式卡片布局。

审核页建议使用三栏：

1. 左栏
   截图、原始文本、关键 DOM 片段。
2. 中栏
   商品结构化结果表单，可逐字段编辑。
3. 右栏
   历史版本、差异结果、发布风险提示。

### 6. 核心工作流

#### 数据源管理

1. 新增数据源。
2. 配置 `crawlMode`、启用状态、抓取限制。
3. 保存后进入任务触发列表。

#### 采集任务

1. 管理员选择数据源并发起抓取。
2. 后端返回任务 ID。
3. 前端轮询任务状态。
4. 进入 `WAITING_HUMAN` 时推送到人工验证队列。
5. 成功后流转到审核发布。

#### 审核发布

1. 审核员查看 AI 结构化结果。
2. 修正价格、库存、标准类型等字段。
3. 确认差异检测结果。
4. 发布到数据仓库。
5. 记录操作者、版本号和 traceId。

#### 回滚

1. 从发布记录中选定历史版本。
2. 查看回滚影响范围。
3. 二次确认后执行回滚。
4. 写入审计日志。

### 7. 数据与接口建议

后台接口建议按职责分组：

1. `/admin/api/sources`
   数据源增删改查。
2. `/admin/api/tasks`
   创建任务、获取状态、重试任务。
3. `/admin/api/manual`
   人工验证接续。
4. `/admin/api/reviews`
   审核查询、保存草稿、发布。
5. `/admin/api/releases`
   发布历史、版本详情、回滚。

统一返回结构继续保持：

1. `code`
2. `message`
3. `data`
4. `traceId`

### 8. 安全设计

后台至少需要以下控制：

1. 登录态校验。
2. 角色权限校验。
3. 发布与回滚二次确认。
4. 关键操作审计日志。
5. 原始 HTML、Cookie、截图只在后台内网可见。

### 9. 实施顺序

建议按下面顺序落地：

1. 独立后台域名和登录鉴权。
2. 数据源页和任务中心真实接 FastAPI。
3. 人工验证队列。
4. 审核发布页。
5. 发布记录和回滚。
